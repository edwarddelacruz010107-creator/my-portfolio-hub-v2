# Portfolio CMS v5.3 — Production Login Fix
## Complete Audit Report & Remediation Guide

---

## Root Cause Summary

Production login fails because **no superadmin user is ever created** on first deploy. Every other piece of the auth stack works correctly. The `preDeployCommand` in `render.yaml` never called `flask create-superadmin`, so the database had zero users, and every login attempt hit `user = None → "Invalid credentials"`.

Secondary issues compound the problem or create future risk:

| Priority | Bug | Symptom |
|----------|-----|---------|
| **CRITICAL-1** | `preDeployCommand` missing `flask create-superadmin` | 100% login failure on fresh deploy |
| **CRITICAL-2** | `ProductionConfig` requires `PAYMONGO_SECRET_KEY` even when `PAYMONGO_ENABLED=false` | App crashes at startup if billing vars not set |
| **CRITICAL-3** | `BaseConfig.SESSION_COOKIE_SECURE` set `True` then immediately overridden to `False` in same class body | Silent double-assignment, fragile config inheritance |
| **HIGH-1** | `cli_create_superadmin` hardcodes `username='superadmin'` and `email='superadmin@portfolio.local'` | Can't customise via env vars; breaks env-var-driven deploys |
| **HIGH-2** | No automatic superadmin bootstrap on startup | A dyno restart after DB wipe leaves no user |
| **MED-1** | No `/superadmin/system-check` diagnostic page | Can't diagnose production issues without Render shell |
| **MED-2** | `_ensure_default_tenant()` only called when `app.debug` | Production may start without a `default` Tenant row |
| **LOW-1** | Superadmin login: `session.pop('tenant_slug', None)` without HMAC re-stamp | Can trigger `TenantGuard` mismatches |

## Step-by-Step Fix Instructions

### Step 1 — Replace config.py

Copy `cms_fixes/config.py` → project root `config.py`.

Key changes:
- Removed the duplicate `SESSION_COOKIE_SECURE = True / False` in `BaseConfig`
- `ProductionConfig.init_app()` only requires `PAYMONGO_*` when `PAYMONGO_ENABLED=true`
- Added `SUPERADMIN_USERNAME` and `SUPERADMIN_EMAIL` config keys (env-var backed)

### Step 2 — Add app/_superadmin_bootstrap.py

Copy `cms_fixes/app/_superadmin_bootstrap.py` → `app/_superadmin_bootstrap.py`

This module is the single source of truth for automatic superadmin creation.

### Step 3 — Add app/superadmin/system_check.py

Copy `cms_fixes/app/superadmin/system_check.py` → `app/superadmin/system_check.py`

No template file needed — it uses an inline Jinja2 template.

### Step 4 — Patch app/__init__.py (4 targeted edits)

**Edit 4a** — Near top of file, add import:
```python
from app._superadmin_bootstrap import auto_bootstrap_superadmin
```

**Edit 4b** — Inside `create_app()`, inside the `with app.app_context():` block,
immediately AFTER the existing `logger.info("Database connection verified at startup")` line:
```python
# Auto-bootstrap superadmin on every startup (idempotent)
if not app.config.get('TESTING'):
    auto_bootstrap_superadmin(app, db)
```

**Edit 4c** — After `app.register_blueprint(superadmin_blueprint, url_prefix='/superadmin')`:
```python
# System-check diagnostic routes
from app.superadmin.system_check import register_system_check
register_system_check(superadmin_blueprint)
```

**Edit 4d** — Replace the body of `cli_create_superadmin()` as specified in
`cms_fixes/PATCH_app_init.py` (Change 4 section).

### Step 5 — Replace render.yaml

Copy `cms_fixes/render.yaml` → `render.yaml` (project root).

Critical change: `preDeployCommand` now includes `flask create-superadmin`.

## Render Environment Variables (Set in Dashboard)

### Always required
```
SECRET_KEY          — auto-generated by Render (generateValue: true in render.yaml)
FERNET_KEY          — generate: python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
CORE_DATABASE_URL   — from your Render PostgreSQL service
TENANT_DATABASE_URL — from your second Render PostgreSQL service (or same DB)
```

### Superadmin bootstrap
```
SUPERADMIN_USERNAME  = superadmin          (or your preferred username)
SUPERADMIN_EMAIL     = you@yourdomain.com  (your real email)
SUPERADMIN_PASSWORD  = <strong-password>   ← SET THIS. Remove after first login.
```

If `SUPERADMIN_PASSWORD` is not set, the preDeployCommand auto-generates a
password and prints it in the Render deploy log. You must capture it before
the log scrolls away. Setting it explicitly is strongly recommended.

### Optional but recommended
```
REDIS_URL           — from Render Redis service (rate limiting, caching)
MAILERSEND_API_KEY  — email delivery
MAILERSEND_FROM_EMAIL
SENTRY_DSN          — error tracking
```

### Billing (only when PAYMONGO_ENABLED=true)
```
PAYMONGO_ENABLED         = true
PAYMONGO_SECRET_KEY      = sk_live_...
PAYMONGO_WEBHOOK_SECRET  = whsec_...
```

### Production — existing database (re-deploy)

```

```

### Production — wrong password entered

```
[ ] Flash: "Invalid credentials." (no stack trace, no detail leak)
[ ] After 5 failures: "Account locked. Try again in 15 minute(s)."
[ ] After lockout expires: correct password works immediately
```

### Production — locked account recovery (no shell needed)

Option A — wait 15 minutes (lockout auto-expires via `AccountLockout.is_locked`)

Option B — set `SUPERADMIN_PASSWORD=<new-password>` in Render env vars,
           trigger a deploy, login immediately with new password, then
           unset `SUPERADMIN_PASSWORD` and rotate via the UI.

Option C — Render shell:
```bash
flask create-superadmin
# or
flask reset-admin-password   # if that CLI command is registered
```

## Authentication Flow Trace (Post-Fix)

```
POST /superadmin/login
  │
  ├─ _handle_login(require_superadmin=True)
  │    ├─ User.query.filter_by(username=...).first()  ← user found (bootstrap guarantees)
  │    ├─ AccountLockout.is_locked(user, db)          ← False (unless locked)
  │    ├─ user.verify_password(form.password.data)    ← check_password_hash()
  │    │    └─ werkzeug bcrypt comparison
  │    ├─ require_superadmin check: user.is_superadmin == True ✓
  │    └─ _complete_login(user, remember, next, default_next)
  │         ├─ AccountLockout.clear_failed_attempts()
  │         ├─ user.last_login = now; db.commit()
  │         ├─ log_activity('login', ...)
  │         ├─ if not user.is_superadmin: stamp_session_tenant(...)  ← skipped for superadmin
  │         ├─ if user.totp_enabled: → redirect to 2FA
  │         └─ login_user(user, remember=remember)
  │              └─ Flask-Login sets session['_user_id'] = user.id
  │                 Cookie: HttpOnly, Secure=True, SameSite=Lax
  │
  └─ redirect → /superadmin/
```

## Security Review — OWASP Alignment

| Control | Status | Notes |
|---------|--------|-------|
| Password hashing (bcrypt) | ✅ | `generate_password_hash` + `check_password_hash` |
| Brute force protection | ✅ | `AccountLockout`: 5 attempts, 15-min lockout |
| CSRF protection | ✅ | Flask-WTF on all POST forms |
| Session cookie flags | ✅ (after fix) | Secure, HttpOnly, SameSite=Lax |
| Session HMAC signing | ✅ | `stamp_session_tenant` with HMAC-SHA256 |
| Credential logging | ✅ | Passwords never logged |
| Tenant isolation | ✅ | `TenantGuard.validate()` on every request |
| Rate limiting | ✅ | Flask-Limiter with Redis fallback |
| HTTPS enforcement | ✅ | `Talisman(force_https=not debug)` |
| CSP headers | ✅ | Custom CSP in `app/__init__.py` |
| XSS protection | ✅ | `X-XSS-Protection`, `X-Content-Type-Options` |
| SQL injection | ✅ | SQLAlchemy ORM + parameterised queries |
| Privilege escalation | ✅ | `@superadmin_required` decorator |
| Secret exposure on diagnostic | ✅ | `system_check.py` redacts all secrets |
| Password in env var | ⚠️ | `SUPERADMIN_PASSWORD` should be removed after first use |

## Migration Steps for Existing Production Database

If your production database already has migrations applied, the fix is
additive — no destructive changes:

```bash
# In Render preDeployCommand (already in the patched render.yaml):
flask db upgrade              # applies any pending migrations; no-ops if at head
flask ensure-tenant-schema    # CREATE TABLE IF NOT EXISTS for tenant tables
flask ensure-default-tenant   # INSERT IF NOT EXISTS for Tenant + Profile rows
flask create-superadmin       # INSERT IF NOT EXISTS for superadmin user
```

All four commands are idempotent. Running them on a database that already has
all tables and users is a safe no-op.
